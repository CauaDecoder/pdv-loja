# Spec 002: Correcoes pos-venda e relatorios confiaveis

## Problem Statement

O PDV atual registra vendas e permite algumas edicoes limitadas de metadados, mas nao oferece uma Correcao pos-venda completa. Quando uma venda finalizada foi registrada com forma de pagamento errada, quantidade errada, produto indevido ou deveria ser cancelada, a loja nao tem um fluxo confiavel para corrigir o erro preservando historico, ajustando estoque e mantendo relatorios financeiros corretos.

Isso cria risco operacional: a venda pode ficar diferente do caixa real, o estoque pode permanecer incorreto e o fechamento financeiro pode incluir movimentacoes que deveriam ter sido anuladas.

## Solution

Implementar suporte backend para Correcao pos-venda com historico preservado. O sistema deve permitir alterar forma de pagamento, alterar quantidade de item, remover item e Cancelar venda inteira. Toda acao sensivel deve registrar responsavel, data/hora e o que mudou.

Cancelar venda nao deve apagar a venda original. Venda cancelada deve ficar fora da movimentacao financeira do periodo, aparecer separada para rastreabilidade e devolver estoque automaticamente. Alteracoes de quantidade ou remocao de item devem aplicar a diferenca no estoque. Relatorios devem refletir o resultado financeiro liquido correto e indicar correcoes quando elas existirem.

Codex implementa esta spec na camada de backend/funcionalidade. O visual/frontend fica para AntiGravity.

## User Stories

1. As an operador de caixa, I want to view finalized sales, so that I can find a sale that needs correction.
2. As an operador de caixa, I want to open the detail of a finalized sale, so that I can understand what was sold before changing anything.
3. As an operador de caixa, I want to alter the payment method of a finalized sale, so that payment mistakes can be corrected.
4. As an operador de caixa, I want to alter payment details, so that card brand, installments, cash received or change can be corrected.
5. As an operador de caixa, I want to alter item quantity in a finalized sale, so that quantity mistakes can be fixed.
6. As an operador de caixa, I want altering item quantity to adjust stock automatically, so that inventory stays accurate.
7. As an operador de caixa, I want to remove an item from a finalized sale, so that mistakenly sold products can be corrected.
8. As an operador de caixa, I want removing an item to return the removed quantity to stock, so that inventory is restored.
9. As an operador de caixa, I want to Cancelar venda, so that a fully wrong sale can be annulled.
10. As an operador de caixa, I want Cancelar venda to return all sold items to stock, so that cancellation does not leave stock wrong.
11. As an operador de caixa, I want cancelled sales to remain visible, so that the sale history is traceable.
12. As an operador de caixa, I want corrections to preserve the original sale history, so that mistakes are not hidden.
13. As a responsavel pela loja, I want every correction to record responsible person, date and time, so that sensitive actions are auditable.
14. As a responsavel pela loja, I want every correction to record before and after values, so that I can understand what changed.
15. As a responsavel pela loja, I want Cancelar venda to be different from deleting a sale, so that historical records remain trustworthy.
16. As a responsavel financeiro, I want cancelled sales excluded from financial movement, so that the period closing reflects only valid movement.
17. As a responsavel financeiro, I want cancelled sales shown in a separate section, so that I can audit what was annulled.
18. As a responsavel financeiro, I want corrected sales to affect the financial summary correctly, so that payment totals and period totals are accurate.
19. As a responsavel financeiro, I want reports to indicate when corrections happened, so that the closing explains differences.
20. As an operador de estoque, I want stock movements generated by corrections to be visible, so that stock history remains understandable.
21. As an operador de estoque, I want cancellation stock movements to reference the original sale, so that the reason for the stock return is clear.
22. As an operador de caixa, I want invalid corrections rejected, so that quantities cannot become negative or inconsistent.
23. As an operador de caixa, I want a sale that is already cancelled not to be corrected again as if it were valid, so that invalid workflows are blocked.
24. As an operador de caixa, I want correction operations to be transactional, so that stock, sale and correction history cannot partially update.
25. As Codex, I want a service facade for Vendas e correcoes, so that UI code does not own correction rules.
26. As AntiGravity, I want backend contracts for sale details, statuses and correction actions, so that the UI can show the right states.
27. As a test maintainer, I want correction behavior tested at service level against SQLite, so that behavior is stable without brittle UI tests.
28. As a future maintainer, I want schema migrations to be explicit, so that existing store data can migrate safely.
29. As a future maintainer, I want existing Venda no caixa registration to keep working, so that the redesign does not regress current sales.
30. As a future maintainer, I want existing import, backup and stock tests to keep passing, so that corrections do not break unrelated flows.

## Implementation Decisions

- Respect ADR 0001: Correcoes pos-venda preserve history instead of overwriting the original sale silently.
- Add or extend persistent state to represent sale status, at minimum distinguishing valid, corrected and cancelled sales.
- Add persistent correction history that records action type, responsible person, timestamp, affected sale, before values and after values.
- Keep Cancelar venda as cancellation/annulment, not deletion.
- A cancelled sale remains queryable but is excluded from financial movement.
- Cancelled sales appear in a separate reporting section.
- Payment corrections affect financial summaries by payment method and total.
- Quantity corrections and item removals adjust stock by the difference between old and new quantities.
- Cancelling a sale returns all sold items to stock.
- Stock movements produced by corrections should reference the originating sale and correction reason.
- Correction operations must be transactional across sale records, correction history and stock movements.
- Add a service facade for Vendas e correcoes above the database layer.
- Prefer high-level service functions as the test seam instead of testing low-level SQL details directly.
- Existing sale registration behavior should remain compatible with the current Venda no caixa flow.
- Existing period concepts remain in place; corrections apply within the period of the original sale unless a future spec defines cross-period accounting.
- Adding a new item to a finalized sale is out of scope. The shop should create a complementary sale instead.
- Authentication, manager approval, roles and passwords are out of scope. The service should accept a responsible person string and persist it.
- Confirmacao de correcao is a UI responsibility, but backend should expose action errors and validation clearly.
- Reports should compute valid financial movement from sale status/corrections rather than assuming every row in the current sales table is financially valid.

## Testing Decisions

- Test behavior through the Vendas e correcoes service facade using a temporary SQLite database.
- Do not assert private implementation details of SQL queries where behavior can be tested through service calls.
- Tests should cover payment correction changing financial totals.
- Tests should cover quantity correction adjusting stock by the difference.
- Tests should cover item removal restoring stock and changing financial total.
- Tests should cover Cancelar venda restoring all stock and excluding the sale from financial movement.
- Tests should cover cancelled sales appearing in a separate cancelled-sales query/report.
- Tests should cover correction history recording responsible person, timestamp/action type and before/after values.
- Tests should cover invalid corrections, such as negative quantity, unknown sale and correction of a cancelled sale.
- Tests should verify current sale registration still reduces stock as before.
- Prior art exists in the current test suite for import, stock rules and backup service behavior; follow that pattern of behavior-focused tests.
- Where reports are file-producing, prefer testing the summary data seam if introduced; otherwise test the service that feeds report generation before testing rendered files.

## Out of Scope

- Visual redesign of the correction screens.
- Authentication, roles, passwords or manager approval.
- Adding new items to a finalized sale.
- Deleting sales from history.
- Cloud backup or external backup integration.
- Fiscal/accounting redesign beyond cancellation, correction and financial summary behavior required here.
- Replacing the current SQLite persistence model.

## Further Notes

- This spec is intended for Codex.
- It depends conceptually on ADR 0001.
- It provides backend contracts consumed by the shared integration spec.
- AntiGravity can use mocks for these services until Codex delivers the service facade.
